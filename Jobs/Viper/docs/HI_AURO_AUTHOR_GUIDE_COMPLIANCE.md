# Viper HiAuRo Author Guide Compliance

`ViperRotationEntry` follows the native ACR entry contract: author is `Kairo`, `UseCustomUi` is false, target job is VPR, and `Build(string settingFolder)` assembles the `Rotation` without heavy combat decisions.

Resolvers are ordered by priority and use `SlotMode.Gcd` for GCD actions and `SlotMode.OffGcd` for abilities. Follow-up abilities, Serpent's Ire, and True North are off-GCD resolvers; Reawaken, Dreadwinder, Rattling Coil, AoE, ranged fallback, and base combo are GCD resolvers.

State access goes through `HiAuRo.Helper`, especially `HiAuRo.Helper.VPRHelper` for VPR action IDs, status IDs, and gauge data. The port should not depend on copied local ID catalogs or old AEAssist job APIs.

Target-circle positional display uses HiAuRo v0.1.90 `HiAuRo.Rendering.PositionalState.Push` from Viper GCD slot builds, so flank/behind requirements are rendered by the framework VFX indicator instead of a job-local overlay.

Every oGCD `Spell` generated by the helper, opener, or trigger hotkeys must carry `SpellType.Ability`. This explicit marker is required for compatibility with HiAuRo slot execution and event records even when the runtime can infer action categories.

Built-in QT controls are registered for Burst, Potion, Hold, and AoE. Job-specific QT IDs stay in `QTKey`, and timeline triggers manipulate `ViperTimelineState` instead of hidden fight-specific logic inside the base loop.
